## 3. 구조

### 3.1 왜 서버가 필요해졌나

v1은 **한 대의 브라우저 안(localStorage)** 에만 저장했습니다. 이 방식은 **데이터가 그 기기 밖으로 나가지 않습니다.**

학생 50명이 각자 기기를 쓰면, 선생님 컴퓨터에는 **다른 49명의 데이터가 존재하지 않습니다.** 현황판이 영원히 빈 화면이 됩니다. 그래서 중간에서 데이터를 보관해 줄 서버(Firebase)가 필요합니다.

```
   학생 A 기기 ─┐
   학생 B 기기 ─┼─→  🔥 Firebase  ←─  선생님 기기 (현황판)
   학생 C 기기 ─┘     (실시간 DB)
      ...50명
```

### 3.2 파일 구조

```
p0715/
├── 제품 요구사항 정의서 (PRD).md
├── 개발 계획서.md              ← 이 문서
├── Firebase 설정 가이드.md      ← 🔴 지금 해야 할 일
│
├── index.html                  시작 페이지 (학생용 / 선생님용 선택)
├── login.html                  이름 입력 로그인
├── student.html                학생 앱 (폰 화면만)
├── parent.html                 선생님 앱 (현황판 + 상세)
│
├── firebase-config.js          ⭐ 직접 채워야 하는 유일한 파일
├── db.js                       Firebase 연결·저장·구독
├── app.js                      공통 로직 (달력·하트·진화·폰 화면)
├── parent.js                   현황판 전용
├── theme.js                    컬러 테마 (PRD 5장)
├── style.css                   애니메이션 (폭죽·펄스·토스트)
│
└── 서버켜기.ps1                로컬 실행용
```

### 3.3 파일을 이렇게 나눈 이유 — **중복 0줄**

학생 앱과 선생님 앱은 **똑같은 폰 화면**을 씁니다. 그냥 복사했다면 달력 버그 하나를 고칠 때마다 **두 파일을 똑같이 고쳐야** 합니다. 초보자가 가장 크게 데는 지점입니다.

그래서:
- 폰 화면 HTML은 `app.js` 안의 **`PHONE_HTML` 한 곳**에만 있습니다
- 두 HTML은 `<div id="phone-mount">` 만 두고, JS가 채웁니다
- Firebase 를 직접 만지는 코드는 **`db.js` 한 곳**에만 있습니다

**달력을 고칠 일이 생기면 `app.js` 한 곳만 고치면 두 앱에 동시에 반영됩니다.**

### 3.4 불러오는 순서 (중요)

```
firebase-config.js  →  db.js  →  app.js  →  parent.js
```

순서가 틀리면 `initDb is not defined` 같은 오류가 납니다. 각 HTML의 맨 아래에 이 순서로 들어 있습니다.

## 4. 데이터 구조

### 4.1 Firebase 에 저장되는 모양

```
jarada/
  students/
    김민준/                     ← 이름으로 만든 열쇠
      name:       "김민준"
      hearts:     24
      stickers:   3
      tasks:      { "2026-07-15": [ {id, text, done}, ... ] }
      celebrated: { "2026-07-15": true }
      updatedAt:  1752537600000
  events/
    김민준/                     ← 선생님 → 학생 알림
      msg: "❤️ 보너스 하트 5개를 받았어요!"
      ts:  1752537600000
```

### 4.2 기기에만 남는 것 (서버에 안 올림)

| 값 | 저장 위치 | 이유 |
| :--- | :--- | :--- |
| 지금 로그인한 사람 | localStorage `jarada-user` | 이 기기의 세션 |
| 소리 On/Off | localStorage `jarada-sound` | 사람마다 다름 |
| 보고 있는 날짜/달 | 메모리만 | ⚠️ **아래 이유가 중요** |

> **왜 보고 있는 날짜를 서버에 안 올리나:** 올리면 **선생님이 달력을 8월로 넘길 때 학생 화면까지 같이 넘어가 버립니다.** 데이터(할 일·하트)만 공유하고 화면 상태는 각자 가져야 합니다.

### 4.3 🔑 가장 중요한 원칙: `render()` 하나로 통일

"체크박스 눌렀는데 하트는 늘었는데 달력 도장이 안 바뀌는" 지옥을 막는 방법은 하나입니다. **무엇이 바뀌든 화면 전체를 다시 그립니다.**

```javascript
function render() {
  renderCalendar();
  renderTodos();
  renderCharacter();
  renderParentPanel();
}

// 모든 동작의 마지막은 항상 이 두 줄
save();      // 서버에 저장
render();    // 화면 전체 다시 그리기
```

서버 데이터가 바뀌면 Firebase 가 알아서 콜백을 불러주고, 거기서도 똑같이 `render()` 를 부릅니다. **들어오는 길이 하나뿐이라 버그가 숨을 곳이 없습니다.**

## 5. PRD에서 확정되지 않은 값

PRD의 수식과 숫자가 **이미지로 삽입**돼 있어 읽을 수 없습니다. 아래 값으로 진행 중입니다. 실제 값이 정해지면 **`app.js` 맨 위 숫자만** 바꾸면 됩니다.

```javascript
const HEART_PER_TASK    = 1;    // FR-2.1 할 일 1개 완료당
const HEART_ALL_CLEAR   = 10;   // FR-2.2 하루 100% 완주 보너스
const HEART_FROM_PARENT = 5;    // FR-3.1 선생님 보너스 하트

const LEVELS = [
  { min: 0,   emoji: '🥚', title: '아기 씨앗' },
  { min: 10,  emoji: '🌱', title: '무럭무럭 새싹' },
  { min: 30,  emoji: '🌿', title: '가지 치는 잎새' },
  { min: 60,  emoji: '🌸', title: '활짝 핀 꽃망울' },
  { min: 100, emoji: '🌳', title: '자라난 수호 나무' },
];
```

**완료율:** `C = (완료한 할 일 수 / 그날 전체 할 일 수) × 100`
→ 할 일이 0개인 날은 `C = 0` 으로 예외 처리돼 있습니다 (0으로 나누기 방지).

## 6. 알려진 한계 — **발표 때 물어보면 이렇게 답하세요**

몰라서 남긴 게 아니라 **기간에 맞춰 내린 결정**입니다. 솔직히 말하는 편이 훨씬 좋은 인상을 줍니다.

### 6.1 이름만 입력하는 건 로그인이 아닙니다

비밀번호가 없으므로 누구나 남의 이름으로 들어갈 수 있습니다. 정확히는 **프로필 선택**입니다.
→ 제대로 하려면: Firebase Authentication (익명 또는 이메일 로그인)

### 6.2 이름이 같으면? → ✅ **해결됨 (번호로 구분)**

로그인 화면에 **번호 칸**이 있습니다. 동명이인이 있을 때만 넣으면 되고, 나머지 학생은 비워둡니다.

- 번호 없음 → 열쇠 `김민준`, 화면 표시 `김민준`
- 번호 7   → 열쇠 `김민준-7`, 화면 표시 `김민준 7번`

이미 있는 이름으로 들어가려 하면 **"본인이 맞나요?"** 하고 한 번 확인합니다. 비밀번호가 없으므로 이게 할 수 있는 최선의 확인입니다.

> **반 명단에 동명이인이 있다면, 그 학생들에게만 번호를 꼭 넣으라고 안내하세요.** 앱이 강제하지는 못합니다.

### 6.3 보안 규칙이 열려 있습니다

`.read: true, .write: true` — 주소만 알면 누구나 전체 데이터를 읽고 지울 수 있습니다.
→ 교실 실습·발표 범위에서는 괜찮지만, **실명과 진짜 개인정보는 넣지 마세요.**
→ 테스트 모드는 **30일 뒤 자동으로 잠깁니다.** 발표가 그 뒤라면 규칙을 다시 게시하세요.

### 6.4 동시에 조작하면 한쪽이 덮어씁니다 (드묾)

**선생님이 하트를 보내는 그 순간(0.1~0.3초 이내)에 학생이 체크를 하면**, 나중에 도착한 쪽이 앞의 것을 덮어써서 하트 1개가 사라질 수 있습니다. 저장할 때 학생 데이터를 통째로 쓰기(`set`) 때문입니다.

발표·실습 범위에서 마주칠 확률은 매우 낮아 그대로 뒀습니다.
→ 제대로 하려면: Firebase 트랜잭션(`transaction()`) 또는 `ServerValue.increment()` 로 하트만 원자적으로 더하기

### 6.5 무료 요금제 한도

| 항목 | 무료 한도 | 학생 50명 | 여유 |
| :--- | :--- | :--- | :--- |
| **동시 접속** | **100명** | 51명 | ✅ 절반 |
| 저장 용량 | 1 GB | 1 MB 미만 | ✅ |
| 월 다운로드 | 10 GB | 여유 | ✅ |

**동시 접속 100명이 유일하게 신경 쓸 한도입니다.** 100명을 넘길 계획이면 그때 다시 상의하세요.

## 7. 초보자를 위한 조언

1. **막히면 F12 → Console 을 먼저 보세요.**
   빨간 글씨가 답을 알려줍니다. `permission_denied` 면 보안 규칙, `Can't determine Firebase Database URL` 이면 Realtime Database 미생성입니다.

2. **화면이 이상하면 `console.log(state)` 를 찍으세요.**
   십중팔구 데이터가 이상한 겁니다. 디버거보다 이게 빠릅니다.

3. **파일을 더블클릭하지 말고 `서버켜기.ps1` 로 여세요.**
   `file://` 로 열면 브라우저가 여러 기능을 막습니다.

4. **한 번에 하나씩 확인하세요.**
   Firebase 설정 → 학생 1명 → 그 다음 선생님 화면. 한꺼번에 하면 어디가 문제인지 못 찾습니다.

5. **완주 조건을 다시 확인하세요.**
   목표는 "PRD 100% 구현"이 아니라 **"남에게 보여줄 수 있는 동작하는 프로그램"** 입니다. 지금 기능은 이미 충분합니다. **남은 건 설정과 배포뿐입니다.**
